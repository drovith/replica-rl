Field guide

How distributed tracing actually works.

One user click can fan out across a dozen services. Tracing stitches those hops back into a single story so you can see where a request spent its time. Here's the model, end to end — no hand-waving.

The vocabulary

Three nouns and you've got it.

trace

The whole journey

One request from edge to edge. Every service it touches contributes to the same trace, identified by a single 128-bit trace_id that rides along for the entire trip.

span

One unit of work

A named, timed operation — an HTTP handler, a SQL query, a Kafka publish. A span records start time, duration, status, and a bag of attributes. Spans nest to form the trace.

context

The thread that connects

The metadata — trace ID and parent span ID — passed between services so a child span knows who its parent was. Propagated over headers using the W3C traceparent standard.

Anatomy of a span

What's actually inside one span.

A span is just structured data. The richer the attributes you attach, the more questions you can answer later — without redeploying to add a log line.

  • →Identity: trace_id, span_id, parent_span_id.
  • →Timing: a start timestamp and a duration, in nanoseconds.
  • →Status: OK, ERROR, or UNSET — the basis of error-rate math.
  • →Attributes & events: arbitrary key/values and timestamped logs scoped to the span.

The pipeline

From a click to a waterfall, in five moves.

This is the exact path a request takes through an instrumented system into Spanforge.

01

A root span opens at the edge

Your API gateway receives POST /v2/checkout. The OTel SDK mints a fresh trace_id and starts a root span. From here, everything that happens belongs to this trace.

02

Context propagates on every hop

When the gateway calls cart-svc over gRPC, the SDK injects a traceparent header. The callee reads it, opens a child span pointing back at its parent, and the chain extends — across processes, languages and the network.

03

Spans close and flush to the collector

As each operation finishes, its span is recorded with a final duration and status, then batched to an OpenTelemetry Collector. The collector is your buffer and routing layer — it never blocks the request path.

04

Spanforge reassembles & samples the trace

We buffer all spans sharing a trace_id until the trace is complete, then run your tail-sampling policy. Errors and slow outliers are kept in full; healthy traffic is sampled to a baseline rate. Only then does anything count against ingest.

05

You read it as a waterfall

The reassembled trace renders as a time-ordered waterfall: parent above child, width proportional to duration. The longest bar on the critical path is almost always your culprit — here, a 189ms locking query.

Context propagation

One header carries the whole trace.

The W3C Trace Context spec defines a single traceparent header. Read its four fields and you can reconstruct the entire parent-child relationship without any shared state between services.

  • →version — 00, the current format.
  • →trace-id — 16 bytes, constant for the whole request.
  • →parent-id — the caller's span ID, becomes your parent.
  • →flags — sampling hints, e.g. 01 = sampled.

Common questions

The things engineers ask first.

Q1

Does instrumentation slow my services down?

Span creation is a few microseconds and export is asynchronous and batched off the request path. In practice the overhead is well under 1% of request latency, and you can cap it with sampling at the SDK if you need to.

Q2

What if one service isn't instrumented?

You get a gap in the waterfall where that span would be. Context still propagates through it as long as the service forwards the traceparent header, so the chain reconnects on the other side.

Q3

Tracing vs. metrics vs. logs — do I need all three?

They answer different questions. Metrics tell you that something is wrong, traces tell you where, logs tell you why. Spanforge links all three by trace ID so you move between them in one click.

Q4

Is this OpenTelemetry, or your own thing?

Pure OpenTelemetry. We ingest OTLP and never ask you to install a proprietary agent. If you leave, your instrumentation comes with you — that's the whole point of the standard.
